# lift-tracker — phone PWA for logging lifts

Single-file offline-first PWA. `index.html` is the whole app (vanilla JS, no build,
no framework). Reads `program.json` (generated by `knowledge/Fitness/tracker/gen_program.py`,
pushed here via `push_program.sh`), writes set/note rows to Supabase via GoTrue + PostgREST
REST (no SDK). `sw.js` caches assets; `program.json` is network-first, everything else
stale-while-revalidate.

## Deploy

Public GitHub Pages repo (`andrewmfoster.github.io/lift-tracker/`). Push to `main` →
`.github/workflows/pages.yml` auto-builds live (~1 min). The Supabase anon key in
`index.html` is public by design — RLS guards the data. Do not commit any service-role key.

## Gotchas

- **Ship = bump the cache, or the phone keeps old code.** `index.html`/`sw.js` are
  stale-while-revalidate, so a code change alone can serve stale for a load or two. On any
  behavior change bump `sw.js` `CACHE` (e.g. `lift-v3`) AND the `BUILD` stamp in
  `index.html`; the SW `activate` handler purges non-current caches. After deploy, close +
  reopen the PWA once to force `activate`.

- **Offline queue: clear entries by identity, not by key.** `flush()` sends queued rows
  then removes them from `queue`. `enqueue()` assigns a *brand-new object* to
  `queue[table:id]` on every edit, so a re-tick during an in-flight POST replaces the entry.
  Deleting blindly by `table:id` after the await drops that newer, never-sent row — the
  original symptom was `done` ticks vanishing while reps/weight (in the sent row) landed, so
  Supabase held `done=false`. Fixed b7192ec: snapshot the exact entry per key and
  ref-compare (`queue[key] === entry`) before deleting. Any future queue rework must keep
  version/identity-checked deletes.
